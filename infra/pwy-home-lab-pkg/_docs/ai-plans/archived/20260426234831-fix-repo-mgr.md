# Plan: Fix fw-repo-mgr ‚Äî CLI args, TypeError bug, _EPHEMERAL, and Make support

## Objective

Fix three distinct problems in `fw-repo-mgr`: (1) CLI uses bare positional subcommands
(`build`, `validate`, `status`) instead of GNU-style flags, violating the CLAUDE.md
convention; (2) running `./fw-repo-mgr build` crashes with `TypeError: 'NoneType' object
is not iterable` when the tier-3 framework config has `framework_repos:` set to null;
(3) `$_EPHEMERAL` is undefined and should be `$_RAMDISK_DIR`. Additionally, add Makefile
support and rename the entry script to `run` per the convention (entry-point named `run`
when a sibling Makefile exists).

## Context

### Root cause of TypeError

The tool resolves `framework_repo_manager.yaml` via a 3-tier lookup. Tier 3 is
`infra/_framework-pkg/_config/_framework_settings/framework_repo_manager.yaml`, which has:

```yaml
framework_repos:

  # Deployment repos belong in the deployment repo's...
```

YAML parses `framework_repos:` followed only by a blank line and comments as `null`. When
this config is used (e.g. if `_FRAMEWORK_MAIN_PACKAGE_DIR` is unset), `_list_repos` outputs
`null` as JSON, and the build-dispatch one-liner `[print(r['name']) for r in json.load(sys.stdin)]`
fails with `TypeError: 'NoneType' object is not iterable` at `File "<string>", line 1`.

Fix: change `framework_repos:` to `framework_repos: []` in tier-3 AND add `or []` guards
in `_list_repos` and the dispatch one-liner.

### The `$_EPHEMERAL` block

Lines 4‚Äì8 of `fw-repo-mgr`:
```bash
python3 "$_UTILITIES_DIR/python/validate-config.py" >&2
_vc_flag="$_CONFIG_TMP_DIR/validate-config-last-run"
if [[ -f "$_vc_flag" ]] && python3 -c "..." 2>/dev/null; then
    bash "$_EPHEMERAL" || true
fi
```

`$_EPHEMERAL` is never defined anywhere. With `set -u`, if the condition is ever true it
would abort the script. The user has confirmed `_EPHEMERAL_DIR` is now `_RAMDISK_DIR`, so
update `$_EPHEMERAL` ‚Üí `$_RAMDISK_DIR`. With `|| true` this becomes a harmless no-op
(bash cannot execute a directory), which is acceptable since the mechanism was never
completed. No other framework tools use this block.

### CLI args ‚Äî current vs desired

Current (violates CLAUDE.md "no positional subcommands"):
```
./fw-repo-mgr build [<name>] [--force-push]
./fw-repo-mgr validate [<name>]
./fw-repo-mgr status
./fw-repo-mgr help
```

Desired (GNU-style):
```
./run -b|--build [<name>] [-f|--force-push]
./run -v|--validate [<name>]
./run -s|--status
./run -h|--help
```

### Rename convention

CLAUDE.md: "name the entry-point script `run` only when a sibling Makefile exists". Since
we are adding a Makefile, rename `fw-repo-mgr` to `run`. Consequences:
- `set_env.sh` line 39: `$_FW_REPO_MGR` must point to the new `run` path
- `set_env.sh` `_set_env_update_path`: remove `_fw-repo-mgr` from PATH (same reason
  `_generate-inventory` is excluded ‚Äî entry-point named `run` is too generic for PATH)

### No callers to update

`grep -r "fw-repo-mgr" infra/` shows no callers that pass subcommand args. The only
consumer is `$_FW_REPO_MGR` (which will be updated). External docs/comments use the old
syntax but do not execute it.
